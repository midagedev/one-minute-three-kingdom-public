# Public Boundary

This repository is a public showcase for One Minute Three Kingdom. Its job is to
help interested players understand the project without exposing live operations,
private implementation, or invite-only community surfaces.

The public boundary is intentionally narrow: transparency should build trust and
curiosity, but not publish material that makes abuse, impersonation, or unwanted
pressure on beta players easier.

## Public In This Repository

The following material belongs here when it is written for public release:

- Project introduction and high-level concept
- Public beta guidance that does not contain invite-only access details
- Screenshots, images, logos, and media cleared for public sharing
- Public-facing dev notes, making-of posts, and design reflections after review
- Community goals, tone, player-facing principles, and non-sensitive FAQs
- High-level roadmap themes that do not reveal hidden systems or operations

## Not Public In This Repository

The following material must stay out of this repository:

- Private source code, build scripts, deployment files, or infrastructure details
- Real server addresses, service topology, tunnel names, or private network names
- Operator accounts, admin workflows, credentials, tokens, secrets, or API keys
- Invite links, invite-only channels, beta roster details, or account identifiers
- Internal operating logs, moderation records, incident notes, or support history
- Detailed abuse-response playbooks, detection rules, thresholds, or enforcement
  timing
- Exact hidden-state values, economy formulas, unreleased system numbers, or
  spoiler-heavy content that would damage play
- Links to private repositories, private task boards, private docs, or internal
  coordination spaces

## Adversarial Review

Public material should be reviewed with a simple hostile-reading question:
"Could this help someone pressure players, impersonate staff, discover private
surfaces, automate abuse, or reverse-engineer hidden game state?"

If the answer is yes, publish a safer summary instead. Prefer high-level intent,
player-facing principles, and redacted examples over raw logs, exact numbers, or
operational mechanics.

This boundary also protects the community goal. A showcase that leaks too much
can turn curiosity into extraction, speculation, or status pressure. A showcase
that says too little can feel opaque and untrustworthy. The useful middle is:
show the game's premise, craft, and public tone while keeping live operations and
player-specific context private.

## Publishing Checklist

Before adding public content, check that it:

- Contains no credentials, private links, account names, server details, or logs
- Does not identify beta players or invite-only spaces
- Does not reveal hidden mechanics beyond approved public explanation
- Does not teach abuse, evasion, scraping, probing, or impersonation
- Still helps a new reader understand the project and its community intent
